このチャプターで見てきたものを、一枚の表にまとめます。

| 箱 | 文脈 | map | flatMap |
| --- | --- | --- | --- |
| `Array<A>` | 複数あるかもしれない | `arr.map(f)` | `arr.flatMap(f)` |
| `Option<A>` | 無いかもしれない | 自作した | 自作した |
| `Either<E, A>` | 失敗の理由付き | 同じ発想で作れる | 同じ発想で作れる |
| `Promise<A>` | あとで届く | `p.then(f)` | `p.then(f)` |

どの行も「文脈付きの値」であり、中身を変換する `map` と、箱を返す関数をつなぐ `flatMap` を持ちます。Promise だけは変則的で、`then` が 1 つで両方を兼ねています(渡した関数が Promise を返すと自動で平らにする)。右のコードで、Array と Option と Promise の 3 つを実際に並べて動かしてみましょう。Promise の行がまだ TODO なので、`then` の連鎖を足してください。

```ts
+++Promise.resolve(10)
  .then((n) => n / 2)
  .then((n) => console.log(n))+++
```

Run すると、3 つの箱が同じ「10 → 半分」の変換をそれぞれの文脈でこなします。Promise の書き方が初めてでも、読めたはずです。`then` を「Promise の flatMap」だと知っていれば、動きは予測できるからです。

## 名前は知識の持ち運び手段

これが、Functor や Monad という名前を学んだ理由です。抽象に名前が付いていると、**1 つの箱で身につけた知識が、他のすべての箱に移植できます**。

- 新しいライブラリで `map` を見かけたら、「箱を開けずに中身を変換するのだな。法則があるから合成もできるな」と、ドキュメントを読む前から当たりが付く
- 「この型は Monad か?」という 1 つの問いで、「逐次処理をつなげるか」「途中の失敗はどうなるか」「リファクタリングは安全か」がまとめて確認できる
- 逆に自分で型を設計するときは、「map と flatMap を法則付きで提供する」だけで、利用者に馴染みの操作一式を約束できる

次の Part で学ぶ Effect も、この表の新しい 1 行にすぎません。文脈は「まだ実行されていない計算」。`Effect.map` と `Effect.flatMap` が何をするかを、あなたはもう知っています。
