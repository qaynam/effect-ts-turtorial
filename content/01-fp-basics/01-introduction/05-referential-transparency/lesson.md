---
title: "参照透過性"
summary: "式を結果で置き換えられる性質"
expectedOutput: |
  直接: 101, 101
  抽出: 101, 101
  置換: 101, 101
---

ここまでのレッスンで、純粋関数と計算をありがたがってきました。その「ありがたさ」の正体に名前を付けます。**参照透過性**(referential transparency)です。ある式を、その評価結果で置き換えてもプログラムの意味が変わらないとき、その式は参照透過だと言います。

たとえば `add(2, 3)` が純粋なら、コードのどこに現れても `5` に置き換えられます。この性質があるおかげで、コードを読むときは頭の中で式を展開できますし、**変数への抽出、共通部分のくくり出し、使わない呼び出しの削除**といったリファクタリングが「意味を変えない」と保証されます。数式を変形するのと同じ要領でコードを推論できるので、これを**等式推論**(equational reasoning)と呼びます。

逆に `Math.random()` や、外部の状態を読み書きする関数の呼び出しは置き換えられません。1 回目と 2 回目で結果が違うのですから、そもそも「評価結果」が 1 つに定まらないのです。

## 等式推論が壊れるところを見る

エディタの `nextTicket` は、呼ぶたびに外側の `issued` を書き換えます。まず Run してください。

同じ `nextTicket()` という式を 2 回書いた (1) と、いかにも安全そうな「変数への抽出」をした (2) とで、結果が食い違います。純粋な世界なら等価なはずの書き換えが、ここでは**プログラムの意味を変えてしまった**わけです。参照透過でない式の上では、等式推論が成り立ちません。

## 参照透過な関数に直す

外側の状態ではなく、引数だけで結果が決まるように書き換えましょう。まず `nextTicket` を、番号を受け取って発券番号を返す `ticketFor` に置き換えます。

```ts
~~~let issued = 0~~~
~~~function nextTicket(): number {~~~
~~~  issued = issued + 1~~~
~~~  return issued~~~
~~~}~~~
+++function ticketFor(customer: number): number {
  return 100 + customer
}+++
```

次に、呼び出し側を新しい関数に合わせます。

```ts
console.log(`直接: ${+++ticketFor(1)+++}, ${+++ticketFor(1)+++}`)

const ticket = +++ticketFor(1)+++
console.log(`抽出: ${ticket}, ${ticket}`)
```

最後に、置き換えのとどめとして、呼び出しそのものを評価結果の `101` に書き換えた行を足します。

```ts
+++console.log(`置換: ${101}, ${101}`)+++
```

Run すると 3 行とも同じ結果になります。式のままでも、変数に抽出しても、答えの数値に置き換えても意味が変わらない。これが参照透過性です。

もっとも、参照透過な式だけではプログラムは完成しません。画面に出す、通信する、時刻を読む、といったアクションが必ず要るからです。Part 3 で学ぶ Effect は、そうした副作用を**実行せずに値として持つ**ことで、副作用を含む処理まで参照透過に扱えるようにする道具です。「計算を厚く、アクションを端に」という前のレッスンの指針が、そこで一気に強力になります。
